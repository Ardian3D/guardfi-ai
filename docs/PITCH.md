# GuardFi AI — Pitch

## 30-second pitch
GuardFi AI is an AI-powered DeFi risk guardian on HashKey Chain. Paste any contract, and it reads the on-chain bytecode, scores the risk with a deterministic engine, explains it with AI, and anchors a tamper-evident proof on-chain that anyone can verify. It's automated risk intelligence — not a formal audit, not financial advice, and no token.

## 60-second pitch
Before you interact with a DeFi contract, you're trusting code you can't read — hidden owner powers, minting, blacklists, upgradeable proxies. Audits are slow; reports are hard to trust.

GuardFi AI does four things in one flow. First, a deterministic risk engine reads the contract's bytecode and metadata on HashKey Chain and produces a reproducible 0–100 score with clear indicators. Second, DeepSeek turns those findings into a plain-language report — constrained so it only explains what the engine found, with a deterministic fallback if AI is unavailable. Third, we hash the report and your own wallet anchors that commitment in our `GuardFiReportRegistry` contract on HashKey Chain. Fourth, anyone can verify the report by ID — no wallet, no payment.

The full report is a premium product settled with HSP through a swappable adapter. It's an MVP, and we're honest about what's real versus mock — but the whole flow works end to end today.

## Judge Q&A preparation

**Is this an audit?**
No. GuardFi AI is automated risk intelligence and best-effort signal detection. It is explicitly **not a formal audit** and does not guarantee the absence of vulnerabilities. Absence of a signal is not proof of safety.

**Why HashKey Chain?**
The registry is deployed on HashKey Chain mainnet, so proofs live on a real, EVM-compatible chain with a public Blockscout explorer. It also aligns the tool with the ecosystem it analyzes and lets us use HSP for premium settlement.

**What goes on-chain?**
Only a minimal commitment: reporter, target, score, `reportHash`, `metadataURI`, timestamp. Nothing else.

**Why not store the full report on-chain?**
Cost, privacy, and practicality. The full report can be large and may include user context. We store a keccak256 hash so integrity is provable while the report stays off-chain — verifiable without being public-by-default.

**How does the AI avoid hallucination?**
The model receives only the structured scan result and is instructed not to invent vulnerabilities, token data, audit status, or financial advice. Output is strict JSON, validated and sanitized (forbidden claims are stripped), and there's a deterministic fallback. The AI explains; it never changes the score.

**Why HSP?**
HSP is the settlement layer for the premium report. We isolate it behind an adapter so the provider can be swapped without touching the app, and payment never affects the on-chain hash.

**What is real vs mock?**
Real: wallet connect, live scanning, deterministic scoring, AI report + fallback, report hashing, wallet-signed on-chain submit to the deployed registry, public verification, dashboard. Mock/MVP: HSP is a mock adapter (real adapter is a skeleton); data is stored in the browser; source verification is `unknown`.

**What are the current limitations?**
Selector detection is best-effort (false positives/negatives; blind to proxy logic). No database yet (browser storage). No IPFS/explorer source lookup yet. Real HSP not wired. Public RPC can rate-limit writes. And, again — not a formal audit or financial advice.
